# Query-Aware GNN 表格檢索實驗報告

> 實驗日期：2026-03-18
> 對應論文：*Weaving the Table Web: Inductive Multi-Table Retrieval via Heterogeneous GNN*
> 對應計畫：大專生計畫 — Query-Aware 消融實驗

---

## 1. 實驗概述

本報告針對 **TWIG (Table-Wise Inductive Graph)** 架構的 **Query-Aware** 擴展進行三組消融實驗，驗證在推論階段將查詢節點 (query node) 加入動態子圖 (dynamic subgraph) 是否能提升表格檢索效能。

### 實驗設計

| 消融實驗 | 說明 | 變項 |
|----------|------|------|
| **Ablation 1** | M0 (固定表示) vs M1 (查詢感知) | 是否加入 query node |
| **Ablation 2** | 子圖大小影響 | k = 5, 10, 20, 50 |
| **Ablation 3** | 查詢連邊設計 | E0–E4（不同 query edge 配置） |

### Query Edge 模式定義

| 模式 | 連邊 | 說明 |
|------|------|------|
| **E0** | 無 | 不加入 Query 節點（等同原始 TWIG baseline） |
| **E1** | Query→Table | 僅連接查詢與候選表格 |
| **E2** | Query→Table + Query→Page | 加入頁面級關聯 |
| **E3** | Query→Table + Query→Column | 加入欄位級關聯 |
| **E4** | Query→Table + Query→Page + Query→Column | 完整查詢連邊 |

### 每個資料集使用的最佳邊配置

| 資料集 | 最佳配置 | 邊類型 |
|--------|---------|--------|
| FeTaQA | A20 | has_column + same_page |
| OTT-QA | A18 | has_column + similar_content |
| MIMO(en) | A19 | has_column + similar_content + shared_column_name |
| MIMO(ch) | A2 | similar_content |
| E2E-WTQ | A2 | similar_content |
| MMQA | A62 | similar_table + has_column + comes_from + same_page + similar_content |

### 關鍵觀察

1. **k=5 和 k=10 是最佳子圖大小**：當 k=5 時，R@10 被上界限制（只能重排 5 個候選），但 MRR 通常最好；k=10 兼顧 R@10 上界與排序品質。
2. **k 增大反而劣化**：k=20 和 k=50 時，噪音節點增加導致隨機初始化的 query edge 權重進一步擾亂排序。MIMO(ch) 在 k=50 時 R@10 僅 0.17（相比 E0 baseline 的 0.56）。
3. **所有 k 值的 MRR 均低於 E0**：查詢感知重排序並未改善任何資料集的精確排序能力。

## 7. 效能劣化根因分析

### 7.1 隨機初始化的 Query Edge 權重

Query-Aware 的核心問題在於 `build_query_aware_model()` 的實作方式：

```python
# evaluate_retrieval_query_aware.py
def build_query_aware_model(base_model, subgraph):
    new_model = DiffusionModel(...)  # 使用子圖的 metadata (含 query 邊)
    new_model.load_state_dict(base_model.state_dict(), strict=False)  # ← 問題所在
```

- `DiffusionModel` 內部使用 `GraphSAGE(...).to_hetero()` 將同構 GNN 轉為異構 GNN
- `to_hetero()` 會為每種 edge type 建立獨立的子模組
- 載入 `strict=False` 時，**query 相關的 edge type 權重完全是隨機初始化的**
- 這些隨機權重在 GNN 的 message passing 過程中注入噪音，而非有用資訊

### 7.2 粗排天花板 (Coarse Ranking Ceiling)

Query-Aware 的重排序只能在 top-k 粗排結果內重新排序：
- 粗排使用 BGE-M3 的餘弦相似度
- 若正確表格不在 top-k 候選中，重排序無論如何都找不到它
- 因此 R@k 的上界 = 粗排的 R@k

### 7.3 每查詢建構新模型的開銷

目前的實作為每個查詢都建構一個新的 `DiffusionModel`，這不僅效率低，更重要的是每次都重新觸發 `to_hetero()` 轉換，生成新的隨機權重。

## 8. 改善建議

若要使 Query-Aware 方法生效，需解決以下問題：

### 8.1 訓練 Query Edge 權重（關鍵）
- 使用 `train_model_query_aware.py` 中的 `QueryAwareDiffusionModel` 進行端到端訓練
- 確保 query edge 的子模組在訓練過程中學到有意義的權重
- 訓練時需大量 query-subgraph 樣本對

### 8.2 固定模型結構
- 預先定義包含 query edge types 的完整 metadata
- 在同一個模型中同時訓練 base edges 和 query edges
- 避免 `strict=False` 載入導致的權重缺失

### 8.3 Cross-Attention 替代方案
- 不使用 GNN message passing 融合 query 資訊
- 改用 cross-attention 機制讓 table embedding 關注 query embedding
- 此方法不需修改圖結構，權重可透過標準反向傳播學習

## 10. 總結

| 發現 | 結論 |
|------|------|
| M1 vs M0 | ❌ Query-Aware (M1) 全面劣於固定表示 (M0)，平均 R@10 ↓7.1%, MRR ↓11.6% |
| 子圖大小 | k=10 最佳平衡點；k 增大引入更多噪音，效能急劇下降 |
| 連邊設計 | E0 (無 query node) 最優；E1–E4 均劣化排序品質 |
| 根因 | Query edge 權重未經訓練（隨機初始化），注入噪音而非有用資訊 |
| 建議 | 需端到端訓練 query edge 權重，或改用 cross-attention 機制 |

**結論：目前的 Query-Aware 實作無法提升效能。核心問題是推論階段動態加入的 query edge 使用隨機初始化權重，這些未經訓練的權重在 GNN message passing 中引入噪音，破壞了原本已學好的表格表示。若要實現有效的查詢感知檢索，必須在訓練階段就加入 query node 並端到端學習 query edge 的權重。**
