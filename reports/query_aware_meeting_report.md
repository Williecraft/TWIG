## 1. 我們在做什麼？

一句話：**使用者問一個問題，我們要從一大堆表格中找出跟這個問題有關的表格。**

比如使用者問「2020年奧運獎牌數前五名的國家？」，我們的系統要從幾千張表格中，找出那張有奧運獎牌資料的表格。

---

## 2. 原本的方法（TWIG）怎麼做的？

### 2.1 把表格變成一張「關係圖」

我們不是把每張表格獨立看待，而是把所有表格建成一張**大圖**，圖上有三種東西（節點）：

- **表格節點**：每張表格一個
- **欄位節點**：每個欄位一個（例如「國家名」、「獎牌數」）
- **頁面節點**：每個來源頁面一個（例如「奧運會維基百科頁面」）

這些節點之間用**六種關係（邊）**連起來：

| 邊 | 白話說明 | 例子 |
|---|---------|------|
| has_column | 這張表格「有」這個欄位 | 奧運表格 → 國家欄位 |
| comes_from | 這張表格「來自」這個頁面 | 奧運表格 → 奧運維基頁面 |
| same_page | 兩張表格在「同一個頁面」上 | 男子獎牌表 ↔ 女子獎牌表 |
| similar_table | 兩張表格「內容相似」 | 2016奧運表 ↔ 2020奧運表 |
| similar_content | 兩個欄位「內容相似」 | 國家名欄 ↔ 參賽國欄 |
| shared_column_name | 兩張表格「有相同的欄位名」 | 都有「Gold」欄位的表格 |

### 2.2 用 GNN 讓表格「互相交流」

建好圖之後，我們用 **GraphSAGE**（一種圖神經網路）讓節點之間交換資訊。

白話來說就是：**每個節點去問它的鄰居「你是什麼？」，然後把鄰居的回答整合進自己的表示裡。**

這個過程做兩輪（2層 GNN）：
- 第一輪：每個表格看看自己的欄位、頁面、相似的表格長什麼樣
- 第二輪：每個表格看看鄰居的鄰居長什麼樣（資訊傳播了兩跳）

做完之後，每張表格就不再只是「自己的內容」，而是融合了周圍鄰居的資訊。例如一張獎牌表格，它的向量裡不只有自己的內容，還包含了「它跟哪些相似的表格在一起」、「它的欄位跟哪些欄位相似」等等。

### 2.3 怎麼找表格？

```
離線（事先做好）：
  全部表格 → GNN → 每張表格得到一個向量

線上（使用者發問時）：
  問題 → BGE-M3 語言模型 → 問題向量
  問題向量 跟 所有表格向量 算相似度
  排序，取最相似的前 10 張
```

### 2.4 TWIG 的問題

**不管使用者問什麼問題，每張表格的向量都一樣。**

打比方：就像一本書在圖書館裡永遠放在同一個位置，不管你來找什麼書，它都在那裡。但理想的情況是——如果你在找食譜，跟食譜有關的書應該「浮上來」；如果你在找歷史，歷史書應該「浮上來」。

TWIG 做不到這件事，因為它產生表格向量的時候根本不知道使用者要問什麼。

### 2.5 TWIG 邊消融實驗

我們實驗了 64 種邊的組合（6 種邊的所有子集），發現：

| 資料集 | 最佳邊組合 | R@10 |
|--------|-----------|------|
| FeTaQA | has_column + comes_from | **0.9900** |
| OTT-QA | has_column + comes_from + same_page + similar_content | **0.9873** |
| E2E-WTQ | has_column + similar_content + shared_column_name | **0.9180** |
| MIMO-EN | has_column + comes_from + same_page | **0.7669** |
| MIMO-CH | has_column + comes_from + same_page + similar_content | **0.7150** |
| MMQA | has_column + comes_from + same_page + similar_content | **0.8043** |

重點發現：**全開 6 種邊不是最好的，選對 2~4 種邊反而更好。** 每個資料集的最佳組合還不一樣。

## 3. 我們的想法：讓 GNN 看到查詢

### 3.1 核心想法

既然 TWIG 的問題是「GNN 看不到查詢」，那我們就**把查詢也放進圖裡，讓它參與 GNN 的資訊交換**。

具體做法：

1. 把查詢文字用 BGE-M3 編碼成一個向量
2. 在圖裡新增一個 **query 節點**
3. 用邊把 query 節點連到圖中的 table、column、page 節點
4. 讓 GNN 跑一遍——這次 query 的資訊也會傳給表格們

這樣出來的表格向量就不再是固定的了——**同一張表格，面對不同的查詢，會產生不同的向量。**

回到剛才的比方：現在每次你走進圖書館說「我要找食譜」，跟食譜有關的書就會自動浮到最前面。

### 3.2 完整流程圖

```
使用者提問："2020年奧運獎牌前五名？"
    │
    ▼
Step 1：粗排（用原本的 TWIG）
    │  全圖 → GNN → 所有表格的靜態向量
    │  問題向量 × 所有表格向量 → 相似度排序
    │  取前 100 名候選表格
    │
    ▼
Step 2：從大圖中「剪」出一個小圖（子圖）
    │  只保留這 100 張候選表格
    │  以及它們相關的欄位、頁面節點
    │
    ▼
Step 3：把 query 節點塞進小圖
    │  query 連到所有 100 張候選表格
    │  query 連到所有相關的頁面和欄位
    │
    ▼
Step 4：在小圖上跑 Query-Aware GNN
    │  query 的資訊傳給表格 → 表格更了解「使用者在問什麼」
    │  表格的資訊也傳給 query → query 更了解「哪些表格跟我有關」
    │  得到：每張表格的「查詢感知」向量
    │
    ▼
Step 5：重新排序
    │  最終分數 = 0.3 × 粗排分數 + 0.7 × 重排序分數
    │  取前 10 名 → 就是最終答案
```

### 3.3 為什麼要兩階段？

直接在全圖（可能有幾千張表格）上塞 query 節點跑 GNN，每個查詢都要跑一次，太慢也太吃記憶體。

所以先用快速的 TWIG 篩出 100 張候選，再對這 100 張做精細的 Query-Aware 重排序。就像考試先用選擇題篩掉大部分人，再對剩下的做面試一樣。

## 4. v1 為什麼失敗？

### 4.1 v1 的做法

v1 的想法很直覺：
1. 載入已經訓練好的 TWIG 模型
2. 直接把 query 節點塞進去
3. 跑 GNN，看結果有沒有變好

### 4.2 問題出在哪？

GNN 裡面有很多「權重矩陣」（$W$），每種邊都有自己的一組。這些權重是 GNN 在訓練過程中學到的——它們決定了「從鄰居收到的資訊要怎麼處理」。

TWIG 訓練的時候，圖裡**沒有 query 節點、沒有 query 邊**，所以 TWIG 的模型裡**根本就沒有處理 query 邊的權重**。

當我們硬把 query 節點塞進去，PyTorch 會自動幫 query 邊生成一組新的權重——但這組權重是**隨機亂數**，完全沒有經過訓練。

### 4.3 用比喻解釋

想像你有一個訓練好的翻譯團隊：
- 英翻中的翻譯員（= has_column 邊的權重）：訓練有素，翻譯得很好
- 日翻中的翻譯員（= similar_content 邊的權重）：訓練有素，翻譯得很好

現在你突然新增一個「火星語翻中文」的需求（= query 邊），隨便找了一個路人來翻（= 隨機初始化的權重）。這個路人翻出來的東西是亂碼，不但沒幫助，還會**污染**其他翻譯員已經翻好的結果。

因為 GNN 在整合資訊時用的是 **MAX 聚合**——取所有來源中最大的值。隨機亂碼中偶爾會有很大的值，這些大值會覆蓋掉原本正確的翻譯結果。

### 4.4 實驗證據

我們做了對比實驗：M0（不塞 query）vs M1（塞 query，但權重是隨機的）

| 資料集 | 不塞 query (M0) | 塞了 query (M1) | 差多少 |
|--------|----------------|-----------------|--------|
| FeTaQA | 0.515 | 0.545 | +0.030（唯一微幅變好） |
| OTT-QA | 0.774 | 0.765 | -0.009 |
| MIMO-EN | 0.656 | 0.554 | **-0.102** |
| MIMO-CH | 0.520 | 0.298 | **-0.222**（最慘，掉了 22%）|
| E2E-WTQ | 0.475 | 0.410 | -0.066 |
| MMQA | 0.654 | 0.597 | -0.057 |

**5 個資料集變差，平均 R@10 下降 7.09%**。這不是「沒幫助」，是「有害」。

結論很明確：**不能用沒訓練過的隨機權重來處理 query 邊的訊息。**

## 5. v2 做了什麼改進？

v2 的核心思路：**不要隨機初始化 query 權重，而是用「零」來初始化，然後慢慢訓練。**

### 5.1 改進一：零初始化（最關鍵的改進）

```python
# 先載入 TWIG 訓練好的權重
model.load_state_dict(pretrained_state, strict=False)

# 把 query 相關的權重全部設為 0
for name, param in model.named_parameters():
    if 'queries' in name:
        nn.init.zeros_(param)
```

**為什麼零比隨機好？**

回到翻譯的比喻：

- **隨機初始化** = 找一個路人來亂翻 → 翻出來的亂碼會污染結果
- **零初始化** = 那個位置暫時**沒有翻譯員** → query 的訊息完全不會傳出去 → 模型的行為跟原本的 TWIG 完全一樣

數學上：如果權重 $W = 0$，那 $W \times \text{query向量} = 0$（零向量）。零向量在 MAX 聚合中不會覆蓋任何東西，所以表格的嵌入不受影響。

然後在訓練過程中，這些權重會**從零開始慢慢學**。只有當模型發現「傳遞 query 資訊確實有助於找對表格」時，權重才會偏離零。如果沒幫助，權重就會留在接近零的位置——不會造成傷害。

**這就是零初始化的精髓：起點安全（等同於 TWIG），只可能變好，不會變差。**

### 5.2 改進二：差異學習率

模型裡有兩組參數：
- **舊參數**（has_column、similar_content 等邊的權重）：TWIG 已經訓練好了，效果不錯
- **新參數**（query 邊的權重）：剛從零開始，什麼都還沒學

如果用同一個學習速度訓練：
- 速度太快 → 舊參數被改壞（就像一個考95分的學生被「重新教育」成60分）
- 速度太慢 → 新參數學不動（0 值幾乎不會動）

解法：**兩組參數用不同的速度學**。

```python
舊參數：學習率 = 0.0001（小步慢走，微調就好）
新參數：學習率 = 0.0005（大步快跑，趕快從零學起）
```

這就像：老員工只需要微調工作方式，新員工需要加速培訓。

### 5.3 改進三：邊過濾

前面邊消融實驗告訴我們，不是所有邊都有用。所以 v2 訓練和評估時，只保留對每個資料集效果較好的邊組合：

```python
BEST_EDGE_CONFIGS = {
    "feta":    ["has_column", "same_page"],
    "ottqa":   ["has_column", "similar_content"],
    "mimo_en": ["has_column", "similar_content", "shared_column_name"],
    "mimo_ch": ["similar_content"],
    "e2ewtq":  ["similar_content"],
    "mmqa":    ["similar_table", "has_column", "comes_from", "same_page", "similar_content"],
}
```

### 5.4 改進四：分數插值

最終分數不是只看 Query-Aware 模型的重排序，而是把 TWIG 粗排分數也混進去：

$$\text{最終分數} = 0.3 \times \text{TWIG粗排分數} + 0.7 \times \text{QA重排序分數}$$

為什麼？因為 TWIG 的粗排雖然沒有 query-aware，但它很穩定。混合起來可以避免 QA 模型偶爾的「失誤」影響太大。

### 5.5 訓練流程白話版

```
對每個訓練查詢：
  1. 用當前模型（不帶梯度）算出所有表格的向量
  2. 粗排：找出跟這個查詢最相似的 30 張表格
  3. 把正確答案也加進去（確保訓練時看得到正確答案）
  4. 建子圖：這 30 張表格 + 相關欄位/頁面 + query 節點
  5. 在子圖上跑 GNN → 得到「查詢感知」的表格向量
  6. 計算 loss：正確表格的分數要比其他表格高
  7. 反向傳播 → 更新權重（舊參數慢慢更新，新參數快快更新）

每隔一段時間：
  重新挖掘「困難負樣本」= 目前模型最容易搞混的錯誤表格

早停法：
  在驗證集上看 R@10，如果連續 5 輪沒進步就停止
```

### 5.6 一句話總結 v1 → v2

| | v1 | v2 |
|--|----|----|
| Query 權重初始化 | 隨機（亂碼） | 零（安全起點） |
| 有沒有訓練 | 沒有，直接推理 | 有，用差異學習率訓練 |
| 結果 | 比不塞 query 更差 | 大幅超越 TWIG 基線 |

## 7. 結果分析

### 7.1 為什麼 v2 比 TWIG 通用版好這麼多？

改善來自**兩個因素的疊加**，不能全歸功於「查詢感知」：

1. **邊過濾**：v2 用了 `BEST_EDGE_CONFIGS`（只留好的邊），而通用 TWIG 是全邊。邊消融實驗已經證明，過濾邊本身就能大幅提升效果。

2. **查詢感知的重排序**：在粗排候選中，讓表格「看見」查詢後重新排序，確實能把一些原本排在後面的正確表格拉上來。

要精確知道「查詢感知」貢獻了多少，需要一個控制實驗：用同樣的邊配置但不塞 query 節點。但從 MIMO-CH 和 E2E-WTQ 超越 TWIG 最佳消融的結果來看，查詢感知確實提供了額外的價值。

### 7.2 為什麼某些資料集 v2 超不過 TWIG 最佳消融？

**這是一個不公平的比較。** 原因有三：

**原因一：TWIG 最佳消融是「事後諸葛」。**

TWIG 最佳邊消融的意思是：我們跑了 64 種組合，然後回頭看哪一種在**測試集**上最好，就報告那個數字。這等於是用答案來選方法——在實際使用中我們不可能知道哪種組合最好。

v2 使用的邊配置是**事先固定的**（基於先前實驗的經驗選擇），不一定是測試集上的最佳。

例如：FeTaQA 的 v2 用的是 A20（has_column + same_page），但 TWIG 測試集最佳是 A24（has_column + comes_from）——邊配置就不一樣。

**原因二：v2 的預訓練模型有「邊配置不匹配」的問題。**

v2 的 GNN 初始權重來自 `model.pt`，它是用全部 6 種邊訓練的。但 v2 訓練時只用了 2~5 種邊。

打比方：一個員工之前在六人團隊中訓練過，現在突然被調到只有兩人的小團隊，他之前習慣的很多工作流程都不適用了——需要重新適應。

TWIG 最佳消融是直接在那個邊配置下**從頭訓練**的，沒有這個適應問題。

**原因三：粗排品質限制了上限。**

v2 的重排序只能在粗排的 Top-100 候選中選。如果正確答案根本不在 Top-100 裡，再怎麼重排也找不回來。

通用 TWIG 模型的粗排品質低於 TWIG 最佳邊消融，所以 v2 的天花板就受限了。

### 7.3 為什麼 MIMO-CH 和 E2E-WTQ 能贏？

這兩個資料集的特點：

- **MIMO-CH**：中文資料集，規模小（100 個測試查詢）。TWIG 最佳消融的 R@10 = 0.7150，本來就不算高 → QA 重排序有更大的改善空間。
- **E2E-WTQ**：測試查詢只有 61 個，但 TWIG 基線 R@10 已達 0.9180。v2 能做到 0.9344，靠的是在粗排候選中精準地把 1~2 個被遺漏的正確表格撈了上來。

### 7.4 MMQA 為什麼差距最大（-11.1%）？

MMQA 是多跳推理資料集，一個查詢可能需要**多張表格**才能回答。v2 的查詢感知機制只是讓 query 和 table 做資訊交換，不擅長處理「需要同時找到多張表格」的情境。加上 MMQA 的圖規模較大，子圖中的 query 訊息容易被稀釋。

### 7.5 v1 vs v2 的本質差異

| 對比項 | v1 | v2 |
|--------|----|----|
| query 權重 | 隨機亂碼 | 從零開始慢慢學 |
| 對 TWIG 基線的影響 | 破壞（加入噪音） | 保底（起點 = TWIG） |
| 有沒有訓練 | 沒有 | 有（15 epochs 微調） |
| 比不塞 query 好嗎？ | 5/6 更差 | 6/6 更好 |

**根本差異**：v1 是直接拿一個「聽不懂話的翻譯員」上場，v2 是先讓翻譯員「閉嘴」（零初始化），然後**訓練他學會翻譯**後再上場。

---

## 8. 結論

### 做了什麼

1. 在 TWIG 的異質圖中**注入查詢節點**，讓表格嵌入能感知查詢內容
2. 設計了**粗排 + 精排的兩階段管線**：TWIG 粗排挑候選 → Query-Aware GNN 在子圖上重排序
3. 用**零初始化 + 差異學習率**解決了新邊類型引入時的參數初始化問題

### 得到什麼

- **v1 失敗**：隨機初始化的 query 權重是噪音，5/6 資料集更差
- **v2 成功**：零初始化 + 訓練後，**6/6 資料集超越通用 TWIG**，平均 R@10 +10.67%
- **v2 在 2 個資料集上超越 TWIG 最佳邊消融**（MIMO-CH +1.0%, E2E-WTQ +1.6%）

### 還沒做到的

- 另外 4 個資料集還沒超越 TWIG 最佳邊消融，主要瓶頸是：
  - 預訓練模型的邊配置不匹配
  - 粗排品質（候選集）限制了重排序的上限
  - GraphSAGE 的聚合方式（min/max/mean）對 query-table 交互的表達能力有限

### 可能的下一步

- 用 **Cross-Attention** 替代 GraphSAGE 做 query-table 交互，讓模型能學到「query 跟每張 table 的具體關聯」，而不是對所有 table 發送相同的訊息
