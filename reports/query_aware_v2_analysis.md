## 1. 研究背景與動機

### 1.1 問題定義

給定一個自然語言查詢 $q$，從多表格語料庫 $\mathcal{T} = \{t_1, t_2, \dots, t_N\}$ 中檢索出最相關的表格。

### 1.2 TWIG 基線系統

TWIG（Table Web with Induced Graphs）使用異質圖神經網路進行表格檢索，其核心流程為：

1. **圖建構**：將表格語料庫建構為異質圖（HeteroData），包含三種節點（table、column、page）和六種邊關係
2. **GNN 編碼**：使用 GraphSAGE + `to_hetero` 將同質 GNN 轉換為異質版本，學習表格的圖結構感知嵌入
3. **相似度檢索**：查詢嵌入（BGE-M3）與 GNN 產生的表格嵌入計算餘弦相似度，排序返回

**關鍵限制**：TWIG 的表格嵌入是靜態的——對所有查詢產生相同的嵌入。查詢資訊完全沒有參與 GNN 的訊息傳遞過程。

### 1.3 研究假設

如果將查詢節點注入圖中，讓查詢資訊通過 GNN 的訊息傳遞機制影響表格嵌入，則可以產生「查詢感知」的動態表格嵌入，從而提升檢索精度。

### 1.4 六種邊類型

| 位元 | 名稱 | 代碼 | 說明 |
|------|------|------|------|
| 5 | similar_table | tt | 表格間嵌入相似度 Top-K |
| 4 | has_column | tc | 表格→欄位（結構邊） |
| 3 | comes_from | tp | 表格→頁面（結構邊） |
| 2 | same_page | sp | 同頁面表格 |
| 1 | similar_content | cc | 欄位間嵌入相似度 Top-K |
| 0 | shared_column_name | sc | 共享相同欄位名稱的表格 |

使用 6-bit 二進位編碼表示邊組合（A0–A63），例如 A20 = 010100 = tc + sp。

## 2. 方法論

### 2.1 整體架構

本研究分為三個階段的實驗：

```
Phase 1: TWIG 邊消融實驗（64 種邊組合 × 6 資料集）
    → 找出每個資料集的最佳邊配置

Phase 2: Query-Aware v1 消融實驗（3 組消融研究）
    → 驗證查詢注入的可行性與邊模式影響

Phase 3: Query-Aware v2 訓練與評估
    → 從預訓練 TWIG 初始化，微調查詢感知模型
```

### 2.2 Query-Aware 查詢邊模式

定義五種查詢邊模式（E0–E4），控制查詢節點與圖中哪些節點類型建立連接：

| 模式 | 查詢連接 | 說明 |
|------|----------|------|
| E0 | 無 | 基線（無查詢節點） |
| E1 | query → table | 查詢僅連接表格 |
| E2 | query → table, page | 查詢連接表格和頁面 |
| E3 | query → table, column | 查詢連接表格和欄位 |
| E4 | query → table, page, column | 查詢連接所有節點類型 |

每種模式都包含雙向邊（正向 + 反向），例如 E1 包含 `(query, queries, table)` 和 `(table, rev_queries, query)`。

### 2.3 Query-Aware v1 設計（初版）

v1 的評估方式：
1. 載入預訓練 TWIG 模型
2. 對每個查詢，動態建立包含查詢節點的子圖
3. 使用 `strict=False` 載入權重（跳過查詢相關模組）
4. 直接進行推理

**v1 的根本問題**：
- 查詢邊的 GNN 參數是**隨機初始化**的（從未訓練）
- 每個查詢都重新建立模型，效率極低
- 隨機初始化的參數在推理時產生雜訊，破壞基線效果

### 2.4 Query-Aware v2 設計（改進版）

v2 解決了 v1 的所有根本問題：

#### 2.4.1 預訓練初始化 + 零初始化查詢參數

```python
# 載入預訓練 TWIG 權重
missing, unexpected = model.load_state_dict(pretrained_state, strict=False)

# 查詢相關參數零初始化（初始行為 = 原始 TWIG）
for name, param in model.named_parameters():
    if any(kw in name for kw in query_keywords):
        nn.init.zeros_(param)
```

**原理**：零初始化確保模型在訓練開始時的行為與原始 TWIG 完全一致。查詢邊的權重為零意味著查詢節點的訊息不會影響其他節點的嵌入，等同於查詢節點不存在。隨著訓練進行，查詢參數逐漸學習有意義的值。

#### 2.4.2 差異學習率

```python
param_groups = [
    {'params': base_params, 'lr': 1e-4},    # 基礎參數（已預訓練）
    {'params': query_params, 'lr': 5e-4},   # 查詢參數（需要從零學起）
]
```

**原理**：基礎參數已經有良好的預訓練值，使用低學習率避免破壞；查詢參數從零開始，需要更高的學習率加速收斂。

#### 2.4.3 Canonical Metadata

```python
def get_canonical_metadata(edge_mode="E4"):
    node_types = ['table', 'column', 'page', 'query']
    edge_types = [
        # 8 種基礎邊 + 6 種查詢邊 = 14 種邊類型
        ('table', 'has_column', 'column'),
        ('query', 'queries', 'table'),
        ('query', 'queries_page', 'page'),
        ('query', 'queries_column', 'column'),
        # ... 及其反向邊
    ]
```

**原理**：PyTorch Geometric 的 `to_hetero` 會根據 metadata 產生對應的 GNN 子模組。如果不同資料集的 metadata 不一致（某些邊缺失），模型結構會不同，導致權重無法跨資料集通用。Canonical metadata 確保所有資料集使用相同的 14 種邊類型結構，缺失的邊用空張量補齊。

#### 2.4.4 訓練流程

```
1. 快取全圖 table embedding（不帶梯度）
2. 對每個查詢：
   a. 粗排：餘弦相似度取 Top-30 候選
   b. 建子圖：候選表格 + 關聯 column/page + query 節點
   c. 子圖 forward：GNN 訊息傳遞（帶梯度）
   d. InfoNCE loss：query vs 子圖中所有表格
3. 定期挖掘困難負樣本（hard negative mining）
4. Early stopping（依驗證集 Recall@10）
```

#### 2.4.5 評估流程（兩階段重排序）

```
1. TWIG 模型全圖 forward → 固定 table embedding
2. 粗排：query embedding × table embedding → Top-100 候選
3. 對每個查詢：
   a. 建子圖（Top-100 候選 + query 節點）
   b. QA 模型子圖 forward → 動態 table embedding
   c. 分數插值：final = α × coarse + (1-α) × rerank
4. 排序返回最終結果
```

### 2.5 `_ensure_query_entries` 機制

```python
def _ensure_query_entries(self, x_dict, edge_index_dict):
    # 確保所有 canonical node types 存在
    for nt in ['table', 'column', 'page', 'query']:
        if nt not in x_dict:
            x_dict[nt] = torch.zeros((1, embed_dim), device=device)
    # 確保所有 14 種 canonical edge types 存在
    for et in canonical_edges:
        if et not in edge_index_dict:
            edge_index_dict[et] = empty  # (2, 0) 空邊
    # 移除非 canonical 邊（如 self_loop）
    edge_index_dict = {k: v for k, v in edge_index_dict.items() if k in canonical_set}
```

**原理**：`to_hetero` 產生的異質 GNN forward 方法會遍歷 metadata 中定義的所有邊類型。如果實際輸入缺少某種邊類型，forward 會報錯。此方法在每次 forward 前補齊缺失的類型（空張量不會影響訊息傳遞結果），確保模型在任何邊配置下都能正常運行。

## 5. 分析與討論

### 5.1 v1 失敗的根本原因

v1 的核心缺陷可以用一句話概括：**在推理時使用未訓練的隨機參數**。

`to_hetero` 將同質 GNN 轉換為異質版本時，會為每種邊類型建立獨立的訊息傳遞子模組。當使用 `strict=False` 載入預訓練權重時：
- 基礎邊（如 `has_column`、`similar_content`）的子模組正確載入預訓練值
- 查詢邊（如 `queries`、`rev_queries`）的子模組保持 PyTorch 預設隨機初始化

在 GNN 的訊息傳遞中，每個節點的新特徵是其鄰居特徵經過線性變換後的聚合。隨機初始化的查詢邊權重會產生隨機的訊息，混入正常的訊息傳遞中，如同在精確的訊號中加入噪音。

```
正常訊息傳遞: h_table = AGG(W_trained · h_neighbors)
加入查詢後:   h_table = AGG(W_trained · h_neighbors, W_random · h_query)
                                                       ↑ 隨機噪音
```

這解釋了為什麼 M1（含查詢節點）在 5/6 資料集上劣於 M0（無查詢節點）。

### 5.2 v2 的改進機制

v2 通過三個關鍵設計解決了 v1 的問題：

#### 5.2.1 零初始化的漸進學習

```
初始狀態（零初始化）:
  h_table = AGG(W_pretrained · h_neighbors, 0 · h_query)
         = AGG(W_pretrained · h_neighbors)  ← 等同於 TWIG

訓練後（學到有意義的值）:
  h_table = AGG(W_pretrained · h_neighbors, W_learned · h_query)
                                              ↑ 有意義的查詢資訊
```

零初始化保證了兩個重要性質：
1. **初始等價性**：訓練開始時模型行為與 TWIG 完全一致
2. **漸進改善**：訓練過程中查詢參數從零開始逐漸學習，只有在確實有助於降低 loss 時才偏離零值

#### 5.2.2 差異學習率的必要性

| 參數組 | 學習率 | 原因 |
|--------|--------|------|
| 基礎參數 | 1e-4 | 已有良好預訓練值，微調即可 |
| 查詢參數 | 5e-4 | 從零開始，需要更大步伐收斂 |

如果使用統一高學習率，基礎參數會被破壞（catastrophic forgetting）；如果使用統一低學習率，查詢參數無法在有限 epoch 內學到有意義的值。

#### 5.2.3 兩階段重排序 + 分數插值

```
final_score = α × coarse_score + (1-α) × rerank_score
            = 0.3 × TWIG分數 + 0.7 × QA分數
```

分數插值保留了 TWIG 粗排的穩定性，同時引入查詢感知的重排序分數。α=0.3 意味著以 QA 重排序為主，但仍保留 30% 的 TWIG 粗排訊號作為正則化。

### 5.3 QA v2 vs TWIG 的差距分析

#### 5.3.1 同邊配置下 v2 為何多數輸？

在 4.3.2 的同邊配置比較中，v2 只在 2/6 資料集勝出。這意味著**加入 query 節點 + 重排序在多數情況下沒有帶來淨增益**，可能原因：

1. **v2 的預訓練模型與邊配置不匹配**：v2 的初始權重來自 `model.pt`（全邊訓練），但微調時只用了 2~5 種邊。部分邊的子模組從未在這個配置下訓練過，需要重新適應。TWIG 消融是直接用該邊配置從頭訓練，邊與權重完全匹配。

2. **GraphSAGE 的聚合對 query 訊息表達力有限**：query 節點對所有 table 發送相同的訊息，用 min/max 聚合時很難區分不同 table 的重要性。

3. **粗排品質限制了上限**：v2 的粗排使用 `model.pt` 在全圖上 forward，品質可能不如用對應邊配置從頭訓練的 TWIG 模型，限制了候選集的 recall ceiling。

#### 5.3.2 v2 vs TWIG 最佳消融為何不公平？

TWIG 最佳消融是從 64 種組合中**事後選出**測試集上最好的，等同在測試集上做超參數搜尋。v2 的邊配置是預先固定的，且**不一定與 TWIG 最佳相同**：

| 資料集 | v2 用的邊 | TWIG 最佳邊 | 一樣嗎？ |
|--------|----------|-----------|---------|
| FeTaQA | A20 (tc+sp) | A24 (tc+tp) | 不同 |
| OTT-QA | A18 (tc+cc) | A16 (tc) | 不同 |
| MIMO-EN | A19 (tc+cc+sc) | A8 (tp) | 完全不同 |
| MIMO-CH | A2 (cc) | A4 (sp) | 不同 |
| E2E-WTQ | A2 (cc) | A2 (cc) | **相同** |
| MMQA | A62 (5邊) | A0 (無邊) | 完全不同 |

只有 E2E-WTQ 的邊配置恰好一致，這也是 v2 能在 E2E-WTQ 上贏 TWIG 最佳的原因之一。

#### 5.3.3 MIMO-CH 和 E2E-WTQ 為何能贏？

這兩個資料集的共同特點：v2 的邊配置與 TWIG 最佳相同或接近（E2E-WTQ 完全相同，MIMO-CH 只差一個邊），減少了邊配置不匹配帶來的劣勢，讓 query-aware 的增益能夠顯現。

### 5.4 要做公平比較需要什麼？

要真正回答「query-aware 有沒有幫助」，需要一個控制實驗：

```
固定同一個邊配置（例如 TWIG 最佳配置）
→ 用該配置從頭訓練 TWIG → TWIG 基線分數
→ 用該 TWIG checkpoint 初始化 v2，同樣的邊配置 → v2 分數
→ 比較：唯一變數 = 有沒有 query 節點
```

目前的比較不滿足這個條件，因為 v2 的預訓練來源（model.pt）和邊配置都與 TWIG 消融不同。

### 5.5 改進方向

1. **使用最佳邊消融模型作為預訓練初始化**：而非通用 model.pt，使用 per-dataset 最佳邊配置的 checkpoint
2. **擴大粗排候選池**：K=100 可能不夠，嘗試 K=200 或 K=500
3. **凍結基礎參數**：完全凍結預訓練參數，僅訓練查詢相關參數，避免基礎模型退化
4. **最佳 α 搜尋**：per-dataset 搜尋最佳分數插值係數
5. **Ensemble**：多種邊配置的 QA 模型 ensemble

## 6. 結論

### 6.1 主要發現

1. **TWIG 邊消融**證實了不同資料集需要不同的邊配置，全邊配置從未是最優解。`has_column` (tc) 和 `similar_content` (cc) 是最普遍重要的邊類型。

2. **Query-Aware v1 失敗**的根本原因是使用未訓練的隨機參數進行推理，隨機噪音破壞了預訓練嵌入的品質。

3. **Query-Aware v2** 通過預訓練初始化 + 零初始化查詢參數 + 差異學習率 + 兩階段重排序，成功解決了 v1 的所有問題：
   - **在同邊配置比較中，MIMO-CH (+4.2%) 和 E2E-WTQ (+1.6%) 超越 TWIG**
   - **在最佳邊消融比較中，同樣在 MIMO-CH (+2.4%) 和 E2E-WTQ (+1.6%) 上勝出**
   - 其餘 4 個資料集略低於 TWIG，主要瓶頸是預訓練模型與邊配置不匹配、GraphSAGE 聚合表達力有限
   - 要做更公平的比較，需要用 TWIG 最佳邊配置的 checkpoint 初始化 v2

### 6.2 技術貢獻

1. 提出了將查詢節點注入異質圖進行查詢感知表格檢索的方法
2. 設計了零初始化 + 差異學習率的微調策略，解決了新邊類型引入時的參數初始化問題
3. 使用 Canonical Metadata 確保模型結構跨資料集一致性
4. 實現了粗排 + 重排序的兩階段檢索管線

### 6.3 實驗規模

- 完成 384 次 TWIG 邊消融訓練 + 評估
- 完成 114 次 Query-Aware 消融評估
- 完成 6 次 QA v2 訓練 + 評估
- 覆蓋 6 個不同語言和領域的表格檢索資料集
